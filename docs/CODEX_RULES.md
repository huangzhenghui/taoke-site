# Codex 开发规则

## 总原则

后续开发必须围绕 MVP 前进。每次任务都要小步修改、明确验证、避免顺手扩展。

如果用户没有明确要求进入新阶段，不实现返利、支付、订单、分销、社区、自动采集等超出 MVP 的功能。

## 修改范围

每次开发任务必须控制修改范围。

规则：

- 不要一次性改太多文件。
- 优先完成一个稳定的小任务，再进入下一个任务。
- 修改前先阅读相关文件和现有模式。
- 不做无关重构。
- 不调整与当前任务无关的格式、命名、目录。
- 如果发现已有未提交改动，必须保留用户改动，不得擅自回滚。

## 技术选择

遵守项目既定技术栈：

- Next.js
- TypeScript
- PostgreSQL
- Prisma
- Redis
- Docker Compose
- Nginx

规则：

- 不要随意引入新框架。
- 不要为了小功能引入大型依赖。
- 优先使用项目已有工具和模式。
- 引入新依赖前必须说明原因、替代方案和影响范围。
- 外部接口 SDK 不能直接散落在页面或业务模块中。

## 外部接口规则

淘客相关接口必须通过 adapter 抽象。

规则：

- 不要把外部接口地址、参数、字段写死在页面中。
- 不要直接接真实接口，先用 mock adapter。
- 不要在业务页面里直接调用轻淘客、阿里妈妈或淘宝联盟接口。
- 不要把 API key、secret、pid、adzone 等敏感配置写进代码。
- 所有敏感配置必须从环境变量读取。
- 第三方返回数据必须转换成内部类型后再使用。
- 真实接口接入前，先定义接口类型、mock 数据和失败处理策略。

推荐依赖方向：

```text
页面
-> 业务模块
-> 内部服务接口
-> adapter
-> 第三方 API 或 mock
```

禁止依赖方向：

```text
页面
-> 第三方 API
```

## MVP 边界

第一阶段只做：

- 首页。
- 文章页。
- 商品页。
- 分类页。
- 专题页。
- 简单后台。
- SEO 基础能力。
- mock 淘客接口适配层。

第一阶段不做：

- 返利系统。
- 用户分销。
- 支付。
- 购物车。
- 订单中心。
- 用户提现。
- 多角色复杂权限。
- 评论社区。
- 自动采集。
- AI 自动写稿。
- 小程序和 App。

如果一个需求看起来会扩展到上述范围，必须先提醒用户确认阶段边界。

## 每次任务完成后的说明

每次完成任务后，必须说明：

- 修改了哪些文件。
- 每个文件的修改目的。
- 如何运行或验证。
- 是否有未完成事项。
- 是否建议现在执行 git commit。

说明要具体，不要只写“已完成”。

示例格式：

```text
本次修改：
- src/modules/articles/...：增加文章查询逻辑。
- src/app/(site)/articles/[slug]/...：接入文章详情页渲染。

验证方法：
- npm run lint
- npm run test
- npm run dev 后访问 /articles/example

建议：
- 当前任务已形成稳定小步，可以执行 git commit。
```

## 验证规则

每次代码修改都要尽量给出可执行验证方法。

根据任务类型选择：

- 类型检查：`npm run typecheck`
- 代码检查：`npm run lint`
- 单元测试：`npm run test`
- 构建验证：`npm run build`
- 数据库迁移验证：`npx prisma migrate dev`
- 本地页面验证：`npm run dev` 后访问对应页面

如果当前项目还没有相关命令，必须说明“暂时没有该命令”，不能假装已经执行。

## 数据和配置规则

规则：

- 数据库 schema 修改必须通过 Prisma migration。
- `.env` 只提供示例字段说明，不提交真实密钥。
- 提供 `.env.example` 时只能写占位值。
- mock 数据可以提交，但必须放在清晰的 mock 目录或测试目录。
- 生产配置、测试配置、本地配置要分清楚。

## 后台开发规则

后台第一阶段只做内容维护。

规则：

- 后台写操作必须做服务端校验。
- 不只依赖前端表单校验。
- 发布、下线、删除等操作要有明确状态字段。
- 删除优先考虑软删除或下线，避免误删 SEO 页面。
- 后台页面不追求复杂装修，但必须清晰、稳定、可用。

## SEO 开发规则

前台页面开发必须考虑百度 SEO。

规则：

- 主要内容必须服务端可渲染。
- 页面必须有明确 H1。
- 页面必须有 title 和 description。
- URL 必须稳定。
- slug 不能随意变化。
- 重要页面需要面包屑和内链。
- 下线内容要明确返回 404、410 或跳转策略，不能静默展示空页面。

## Git 协作规则

每完成一个稳定小任务，必须提醒用户执行 git commit。

规则：

- 不在用户未要求时自动提交。
- 提交前说明建议提交的原因。
- 如果用户要求提交，再执行 git add 和 git commit。
- 不把多个不相关任务混在一个提交里。
- 不回滚用户已有改动。

## 沟通规则

开发过程中要保持透明。

规则：

- 开始修改前说明将要改哪些范围。
- 遇到范围膨胀时先停下来说明。
- 遇到技术取舍时给出推荐方案和原因。
- 任务结束后等待用户确认，不擅自继续进入下一阶段。
